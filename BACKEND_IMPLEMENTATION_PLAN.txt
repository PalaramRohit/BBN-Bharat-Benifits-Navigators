  name: str
  description: str
  ministry: str
  eligibility_criteria: {
    min_age: int,

6. AGENT ARCHITECTURE & ORCHESTRATION
================================================================================

BASE AGENT CLASS:
```
class BaseAgent:
  - agent_id: str
  - agent_type: str (eligibility, explanation, etc)
  - execute(query, context, user_profile) -> AgentResponse
  - validate_input(query) -> bool
```

CORE AGENTS (MVP - 24 HOURS):

1. ELIGIBILITY AGENT (PRIORITY: HIGH)
   Input: User profile
   Output: List of eligible policies with match scores
   Logic:
     - Rule-based matching against eligibility_rules.json
     - Score calculation based on criteria match
     - Caching of results
   Implementation: Python rules engine + MongoDB queries
   Time: 4-5 hours

2. POLICY EXPLANATION AGENT (PRIORITY: HIGH)
   Input: Policy ID, user language preference
   Output: Simplified policy explanation + documents needed
   Logic:
     - Retrieve policy from database
     - Use LLM to generate simple explanation
     - Extract required documents
     - Return in local language (mock for Hindi)
   Implementation: OpenAI API + prompt engineering
   Time: 3-4 hours

3. GUIDANCE AGENT (PRIORITY: MEDIUM - Skeleton)
   Input: Policy ID, User profile
   Output: Step-by-step claim process
   Status: Skeleton with mock responses
   Implementation: Return hardcoded claims steps for demo
   Time: 2-3 hours

4. OPTIMIZATION AGENT (PRIORITY: LOW - Skeleton)
   Input: User profile, list of eligible policies
   Output: Best combination of benefits
   Status: Skeleton implementation
   Implementation: Return top 3 policies for demo
   Time: 1-2 hours

5. VOICE AGENT (PRIORITY: VERY LOW - Not Started)
   Status: Skeleton only (not functional for hackathon)

ORCHESTRATOR (PRIORITY: CRITICAL):
```
class BharatOrchestrator:
  - route_query(query, user_profile) -> selected_agent
  - execute_workflow(workflow_name, context) -> final_response
  - manage_session(session_id) -> session_state
  - aggregate_responses(agent_outputs) -> unified_response
  - handle_context_switching() -> maintain state across agents
```

Orchestration Logic:
1. User submits query
2. Orchestrator analyzes intent (eligibility? explanation? claim?)
3. Routes to appropriate agent(s)
4. Manages session state
5. Collects outputs
6. Formats unified response
7. Returns to frontend

Workflow Example:
```
User Query: "I'm self-employed with в‚№20k monthly income. What benefits can I get?"

Step 1: Parse profile from query
  в†’ age, income, occupation extracted

Step 2: Route to Eligibility Agent
  в†’ Check against 20+ policies
  в†’ Return 5 eligible schemes

Step 3: Route to Optimization Agent
  в†’ Rank by benefit amount
  в†’ Return top 3

Step 4: Route to Explanation Agent (background)
  в†’ Generate simple summaries for top 3
  в†’ Extract required documents

Step 5: Aggregate & Return
  в†’ Final response with eligible schemes + explanations + documents
```
