1. MULTI-LLM STRATEGY OVERVIEW
================================================================================

AVAILABLE LLMs:
1. Gemini (Google) - Fast, structured outputs
2. Llama 80B (Meta) - Conversational, good for NLU
3. Claude (Anthropic) - Best for step-by-step guidance

STRATEGY: Intelligent orchestration routes queries to the BEST LLM for each task
BENEFIT: Optimized accuracy, speed, cost-efficiency for different agent types

================================================================================
2. Agent → LLM MAPPING MATRIX
================================================================================

AGENT NAME: Policy Explanation Agent
LLM CHOICE: Gemini (Primary)
WHY: 
  - Best at converting complex policy text into simple language
  - Fast API response (critical for demo)
  - Structured output formatting
  - Free tier available
USE CASE: "What are the benefits of PMJAY?"
PROMPT TEMPLATE:
  "Explain this government policy in simple terms for a non-technical user:
   {policy_text}
   Include: Benefits, Eligibility, Required Documents, Deadline
   Format: Bullet points, easy to understand"
RESPONSE: Simple explanation + required docs list

---

AGENT NAME: Claim Guidance Agent
LLM CHOICE: Claude (Primary)
WHY:
  - Excellent at generating step-by-step procedures
  - Good at handling nuanced instructions
  - Strong reasoning for complex workflows
  - Best for claim simulation
USE CASE: "How do I claim health benefits?"
PROMPT TEMPLATE:
  "Generate a step-by-step claim guide:
   Policy: {policy_name}
   User Profile: {user_info}
   Output: Numbered steps with details, documents needed at each step,
           estimated timeline, common mistakes to avoid"
RESPONSE: Detailed claim workflow with steps

---

AGENT NAME: Eligibility Intelligence Agent
LLM CHOICE: RULE ENGINE (No LLM)
WHY:
  - Eligibility must be deterministic & explainable
  - Rules are clear: income limit, age, etc.
  - LLM would introduce unnecessary ambiguity
  - Fast execution needed
USE CASE: "Am I eligible for PM-KISAN?"
LOGIC: Evaluate against eligibility_rules.json
RESPONSE: Yes/No + reason + match score

---

AGENT NAME: Voice/Conversational Agent
LLM CHOICE: Llama 80B (Primary)
WHY:
  - Excellent for natural language understanding
  - Can be run locally (no API latency)
  - Good for extracting intent from casual queries
  - Conversational tone
USE CASE: "Hey, I heard about some farmer scheme, which one?"
PROMPT TEMPLATE:
  "Extract intent and entities from user query:
   Query: {user_query}
   Output: Intent (eligibility/explanation/claim/search), 
           Key entities (scheme name, user type, etc),
           Confidence score"
RESPONSE: Parsed intent + entities → Route to another agent

---

AGENT NAME: Benefit Optimization Agent
LLM CHOICE: Gemini (for ranking explanation)
WHY:
  - Explain why certain benefit combination is best
  - Quick reasoning over multiple policies
USE CASE: "What's the best combination of benefits for me?"
LOGIC: Rule-based scoring + Gemini explanation
RESPONSE: Top 3 benefit combinations with explanations

6. RAG (RETRIEVAL-AUGMENTED GENERATION) SYSTEM
================================================================================

WHAT IS RAG?
Retrieve relevant information from database BEFORE asking LLM to answer
This ensures LLM generates answers based on actual policy data (not hallucinations)

FLOW:
User Query → Embed Query → Search Similar Policies → Get Top 5 → 
Feed to LLM with Context → LLM Answers Based on Context

IMPLEMENTATION:

STEP 1: GENERATE EMBEDDINGS (One-time)
  Input: All 100 policies
  Process: Use Gemini Embeddings API
  Output: Each policy gets 768-dimensional embedding vector
  Storage: Store in MongoDB or embeddings_cache.pkl
  
  Code pseudocode:
  ```
  for policy in all_policies:
  ```

7. LLM SELECTION DECISION TREE
================================================================================

User Query Arrives
    │
    ├─ Is it eligibility-based?
    │  └─ YES → ELIGIBILITY AGENT (Rule Engine, no LLM)
    │
    ├─ Is it asking "HOW" (procedural)?
    │  └─ YES → CLAIM GUIDANCE AGENT → CLAUDE LLM
    │              (Best for step-by-step instructions)
    │
    ├─ Is it asking "WHAT" or "EXPLAIN" (conceptual)?
    │  └─ YES → POLICY EXPLANATION AGENT → GEMINI LLM
    │              (Best for converting complex → simple)
    │
    ├─ Is it vague/conversational?
    │  └─ YES → VOICE/CONVERSATIONAL AGENT → LLAMA 80B
    │              (Best for NLU and casual language)
    │
    └─ Is it optimization/comparison?
       └─ YES → BENEFIT OPTIMIZATION AGENT → GEMINI LLM
                   (Best for ranking and explanations)

8. LLM FALLBACK MECHANISM
================================================================================

PRIMARY LLM FAILS → FALLBACK CHAIN

Scenario: Gemini API timeout

Flow:
1. Try GEMINI → Timeout after 5 seconds
2. Log error: "Gemini unavailable"
3. Fallback to CLAUDE → Use same prompt
4. If CLAUDE succeeds → Return response + log fallback
5. If CLAUDE fails → Fallback to LLAMA 80B (local)
6. If all fail → Return cached/template response

Code Structure:
```python
def execute_policy_explanation(query):
    try:
        response = gemini_api.explain(query)
        return response
    except TimeoutError:
        logger.warning("Gemini timeout, falling back to Claude")
        try:
            response = claude_api.explain(query)
            return response
        except Exception:
            logger.warning("Claude failed, using Llama")
            response = llama_local.explain(query)
            return response
    except Exception as e:
        logger.error(f"All LLMs failed: {e}")
        return get_template_response(query)
```

BENEFITS:
✓ Reliability (if one LLM down, another takes over)
✓ Cost optimization (try faster/cheaper first)
✓ No single point of failure
✓ Automatic degradation (graceful)
